# v0.1 Scope and Boundary

This document defines the scope, non-goals, adoption boundary, review expectations,
and non-canon posture for v0.1 of the Agent Runtime Governance candidate namespace.

It is a candidate document, not an adopted standard.

## Scope

v0.1 is intentionally narrow. It covers the smallest useful, reviewable unit of
runtime governance for agents: a single versioned **governance packet** that can be
authored, reviewed, validated, and referenced during agent execution.

In scope for v0.1:

- A definition of runtime governance for agents as version-controlled source material.
- A minimal, machine-readable schema for a governance packet.
- At least one valid example and one invalid fixture demonstrating validation intent.
- A prior-art survey of public policy engines and adjacent ecosystem primitives.
- A receipt template for recording that a packet was reviewed.
- Clear separation between candidate material and adopted guidance.

Out of scope for v0.1 (deferred to later versions):

- A universal standard or cross-vendor specification.
- Runtime enforcement code, agents, or host integrations.
- Cryptographic signing, key management, or identity binding.
- Automated policy evaluation engines shipped from this repo.
- Adoption criteria, conformance suites, or certification paths.

## Non-goals

- This repo is not a universal standard.
- This repo does not claim ownership of the broader field or its terminology.
- This repo does not canonize any vendor, product, or internal concept.
- This repo must not include private, internal, or secret operational content.
- v0.1 does not define binding authority; it defines a shape that authority could
  later be recorded in.

## Adoption boundary

v0.1 material is **candidate**, not adopted. Nothing in v0.1 should be treated as:

- A normative specification.
- A compliance requirement.
- An endorsement of any vendor, product, or internal system.
- A replacement for existing policy engines or admission controllers.

Adoption of any v0.1 artifact would require a separate, reviewed governance path
that does not yet exist in this repository. Until that path exists, all v0.1
artifacts are exploratory.

## Review expectations

v0.1 artifacts are expected to be reviewed for:

- Clarity of scope and boundary.
- Absence of private, internal, or secret content.
- Correct attribution of prior art and adjacent ecosystem references.
- Minimal, reviewable size with no overclaims of novelty or ownership.
- A working schema that validates the included example and rejects the included
  invalid fixture.

Reviewers should not interpret v0.1 as a commitment to future stability. Field
names, structure, and file layout may change in subsequent versions without
preserving backward compatibility.

## Non-canon posture

This repository is a public seed. It is explicitly **non-canon**:

- No content here is asserted as authoritative for the broader field.
- Terminology is borrowed from public prior art and is not owned by this repo.
- References to external projects are pointers to public prior art, not endorsements.
- Any resemblance to internal or proprietary systems is incidental and is not
  carried forward as canon unless separately reviewed and adopted.

## Required packet pieces

A v0.1 governance packet must contain the following pieces, validated against
`schemas/agent-runtime-governance-v0.1.json`:

1. **schemaVersion** — the schema version this packet conforms to (`v0.1`).
2. **packetStatus** — lifecycle status of the packet (e.g. `candidate`, `adopted`,
   `deprecated`). v0.1 packets should default to `candidate`.
3. **governanceManifest** — identifying metadata for the packet:
   - `id` — stable identifier for the packet.
   - `title` — human-readable title.
   - `version` — packet version (semver-ish string).
   - `scope` — short description of what the packet governs.
4. **authority** — who or what grants authority for this packet and the basis for
   that authority. v0.1 only records the claim; it does not verify it.
5. **governanceContract** — the enforceable shape of governance:
   - `policyEngine` — reference to the policy engine or evaluation approach.
   - `enforcementPoints` — where governance is evaluated in the runtime path.
   - `guardrailSpecs` — guardrail definitions referenced by the packet.
   - `killSwitchPolicy` — when and how a kill switch may be triggered.
   - `circuitBreakerPolicy` — when and how a circuit breaker may open/close.
   - `auditRequirements` — what must be recorded for auditability.
6. **receiptRequirements** — what a receipt for this packet must contain when the
   packet is reviewed or referenced at runtime.

Packets missing any required piece are invalid by design; see
`fixtures/invalid/governance-packet-v0.1.invalid.json`.
